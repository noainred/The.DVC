## 🔴 CRITICAL (4)

### C1. 전역 TLS 검증 비활성화가 프로세스 전체 fetch에 적용(setGlobalDispatcher) → 업그레이드/원격 fetch MITM
- **분류**: security · **확신도**: high
- **위치**: `server/src/vcenter/restClient.js:25-45`
- **문제**: (SECURITY-AUDIT C2 기존, 미수정) config.rejectUnauthorized=false(자체서명 vCenter 환경에서 일반적)이면 `setGlobalDispatcher(new Agent({connect:{rejectUnauthorized:false, ciphers:'DEFAULT@SECLEVEL=0', minVersion:'TLSv1', SSL_OP_LEGACY...}}))`가 프로세스 전역에 적용된다. 이 전역 dispatcher는 undici 기반 모든 fetch()에 적용되므로 upgrade/fetchPackage.js·upgrade/upgrade.js(downloadArchive/checkRemote)·bundleSource.js의 GitHub/미러 fetch도 인증서 검증 없이 수행된다. 결과적으로 versions.json·번들 다운로드 경로가 MITM에 노출되고, 번들은 별도 코드서명이 없어(체크섬도 같은 채널의 versions.json에서 옴) 변조 번들을 받아 applyPackage로 자가설치 → 포탈/에이전트 RCE.
- **수정안**: 전역 setGlobalDispatcher를 쓰지 말고 vCenter 클라이언트 fetch에만 로컬 undici Agent(dispatcher 옵션)를 주입. 업그레이드 fetch는 항상 검증 TLS로 수행. 번들에 Ed25519/minisign 서명을 추가해 설치 전 공개키로 검증.

### C2. /api 라우터 전체에 역할 가드 부재 — viewer가 운영 VM Tools 업그레이드 등 변경 동작 수행
- **분류**: security · **확신도**: high
- **위치**: `server/src/index.js:91, server/src/routes/api.js:1394,1427,1513(/tools/ip-ping),1735,1740`
- **문제**: (SECURITY-AUDIT 기존 C4) index.js에서 `app.use('/api', authMiddleware, api)`로 마운트만 하고 api.js 내부 라우트 전체에 requireRole이 단 한 곳도 없다(grep 결과 requireRole 사용 0건). 따라서 가장 낮은 권한 viewer 토큰만 있으면: POST /api/vms/upgrade-tools(라이브 vCenter에서 UpgradeTools_Task 실행 → 실제 게스트 재부팅/Tools 변경), PUT /api/ui-settings(전 사용자 공유 설정 변경), POST/DELETE /api/alarm-mutes(알람 음소거 조작으로 장애 은폐), PUT /api/tools/ipam/annotation, POST /api/tools/ip-ping(아래 DoS 참고)을 모두 호출할 수 있다. 발동: 유효한 viewer Bearer 토큰으로 해당 엔드포인트 직접 호출. 권한상승.
- **수정안**: 상태변경(POST/PUT/DELETE) 라우트에 requireRole('admin' 또는 'operator')를 개별 부여하거나, 읽기/쓰기 라우터를 분리해 쓰기 라우터 전체에 requireRole를 건다. 특히 vms/upgrade-tools는 admin 전용으로.

### C3. 전역 TLS 검증 비활성화(setGlobalDispatcher)가 프로세스 전체 fetch에 전파 — 업그레이드·LLM·dataplane·collector·NSX 모두 MITM 노출
- **분류**: security · **확신도**: high
- **위치**: `server/src/vcenter/restClient.js:25-45 (config.js:33, store.js:3 경유 기동 시 로드)`
- **문제**: (SECURITY-AUDIT 기존 C2) config.rejectUnauthorized 는 VC_TLS_REJECT_UNAUTHORIZED==='true' 일 때만 켜지므로 기본값은 false. 이 기본 상태에서 restClient.js 최상위의 setGlobalDispatcher(new Agent({connect:{rejectUnauthorized:false, minVersion:'TLSv1', ciphers:'DEFAULT@SECLEVEL=0', secureOptions: SSL_OP_LEGACY_SERVER_CONNECT|SSL_OP_ALLOW_UNSAFE_LEGACY_RENEGOTIATION}})) 가 모듈 import 부수효과로 무조건 실행된다. store.js 가 기동 시 restClient.js 를 import 하므로 dataSource 와 무관하게 프로세스 전역 undici 디스패처가 교체된다. 발동조건: 그 후 dispatcher 옵션을 직접 넘기지 않는 모든 bare fetch() 가 이 허용 디스패처를 상속한다 — 확인된 대상: upgrade/fetchPackage.js, upgrade/upgrade.js, upgrade/bundleSource.js(번들 다운로드), collector/puller.js, collector/upgradePush.js, llm/ollama.js, nsx/client.js(주석은 '전역 디스패처 재사용'이라 명시 — 의도적으로 검증 없음), agent/*Push.js, proxy/dataplane.js. 즉 GitHub/사내미러↔포탈, 중앙↔에이전트, 포탈↔LLM/HAProxy 경로가 전부 인증서 검증·SAN 매칭·최신 TLS·SECLEVEL 보호를 잃어 능동 MITM(ARP/DNS/BGP/경유 HAProxy)으로 응답 위변조가 가능하다. vCenter 가 RTT 800ms 인 폴란드/미동부 경유 구간을 지나가는 만큼 경로상 중계 노드가 많다.
- **수정안**: 전역 setGlobalDispatcher 제거. vCenter/iDRAC 등 자체서명 자산 전용 permissive Agent 를 만들어 해당 fetch 의 dispatcher 옵션으로만 주입한다(이미 redfish.js/ome.js/soapClient 패턴 존재). 업그레이드·LLM·collector·NSX·dataplane fetch 는 기본(검증 ON) 디스패처를 쓰게 하고, 자산별 신뢰(핀/사설CA)는 명시 opt-in 으로만 허용. rejectUnauthorized 기본값을 true 로 뒤집고 자체서명 자산은 thumbprint 핀닝으로 대체.

### C4. 원격 업그레이드 번들에 서명·체크섬 검증 부재 — versions.json 과 동일 채널 신뢰로 트로이목마 자가설치(RCE)
- **분류**: security · **확신도**: high
- **위치**: `server/src/upgrade/upgrade.js:280-352 (downloadArchive/upgradeFromRemote), server/src/upgrade/bundleSource.js:28-39`
- **문제**: (SECURITY-AUDIT 기존 C2 보강) upgrade.js 의 checkRemote→downloadArchive→upgradeFromRemote 경로는 다운로드 파일에 대해 (1)파일명 정규식 ARCHIVE_RE 일치와 (2)MAX_BUNDLE_BYTES 크기 상한만 검사하고 SHA-256/서명 검증이 전혀 없다. bundleSource.js resolveBundleBytes 의 원격 분기(res.arrayBuffer())도 무검증이다. fetchPackage.js downloadPackage 는 SHA-256 을 검증하지만 그 sha 값조차 versions.json(공격자가 MITM 으로 위조 가능한 동일 출처)에서 오므로 무결성 근거가 되지 못한다. 발동조건: 첫 finding 의 전역 TLS 무검증과 결합하면, MITM 공격자가 versions.json 의 latest/해시/tar_gz 와 번들 바이트를 동시에 바꿔 위조 번들을 내려보낼 수 있고, applyPackage 가 이를 설치 디렉터리에 스왑한 뒤 restartProcess 로 재기동 → 포탈 및 푸시받는 모든 collector 에이전트에서 임의 코드 실행. UPGRADE_AUTO_APPLY=true 면 운영자 개입 없이 자동 감염된다.
- **수정안**: 릴리스 번들에 Ed25519/minisign 분리서명을 부착하고, 공개키를 앱에 하드코딩(또는 별도 신뢰루트). 다운로드 후 압축해제 전 서명 검증 통과 시에만 applyPackage 진행. 최소한 SHA-256 을 검증된 TLS 채널(핀닝된 호스트)에서만 받은 versions.json 으로 강제하고, downloadArchive/resolveBundleBytes 에도 동일 검증 추가.

## 🟠 HIGH (21)

### H1. 로그인 잠금(loginRateLimit) 우회 — X-Forwarded-For 스푸핑으로 계정별 잠금 무력화
- **분류**: security · **확신도**: high
- **위치**: `server/src/routes/auth.js:25, server/src/security/loginRateLimit.js:19,35,45`
- **문제**: 로그인 IP를 클라이언트가 보내는 X-Forwarded-For 헤더에서 그대로 취한다(auth.js:25: req.headers['x-forwarded-for']). loginRateLimit는 잠금 키를 keyOf(ip, username)=`${ip}|${username}`로만 만든다(loginRateLimit.js:19). 따라서 공격자가 매 요청마다 X-Forwarded-For 값을 1.1.1.1, 1.1.1.2 ... 로 회전시키면 각 시도가 새 키가 되어 카운터가 누적되지 않고, 같은 계정(admin)에 대해 무제한 무차별 대입이 가능하다. 계정 단독(IP 무관) 카운터가 없어 per-account 잠금이 전혀 걸리지 않는다. 발동조건: 인증 활성(기본값) + 포탈이 인터넷/내부망에 직접 노출(리버스프록시가 XFF를 덮어쓰지 않는 구성). 기본 admin123 시드와 결합 시 즉시 탈취 가능.
- **수정안**: 신뢰하는 프록시 홉 수를 app.set('trust proxy', n)로 설정하고 req.ip를 사용(임의 XFF 무시). 동시에 username 단독 키 카운터를 추가해 IP가 바뀌어도 계정별 누적 잠금이 걸리게 한다(예: keyOf(ip,user)와 keyOf('*',user) 둘 다 증가, 둘 중 하나라도 임계 초과면 차단).

### H2. /api 라우터 역할 가드 부재 — viewer가 운영 VM Tools 업그레이드 등 변경작업 수행
- **분류**: security · **확신도**: high
- **위치**: `server/src/index.js:91, server/src/routes/api.js:1394,1427,1735,1740`
- **문제**: (SECURITY-AUDIT 기존 C4, 미수정) app.use('/api', authMiddleware, api)는 인증만 강제하고 역할은 검사하지 않는다(index.js:91). api.js에는 requireRole import도 사용도 0건이다(`from '../auth'` import 없음). 그 결과 가장 낮은 권한인 viewer 토큰만으로 상태변경 라우트를 호출할 수 있다: POST /api/vms/upgrade-tools(api.js:1394)는 실 vCenter에서 UpgradeTools_Task를 실제로 실행하고(mock이 아니면 upgradeVmTools 호출, 1416), PUT /api/ui-settings(1427)는 전역 UI설정 변경, POST/DELETE /api/alarm-mutes(1735,1740)는 알람 뮤트 생성/삭제. 발동조건: 인증 활성 상태에서 viewer/operator 계정 토큰 보유. 권한상승으로 운영 인프라에 부수효과를 일으킨다.
- **수정안**: 상태변경 라우트에 requireRole('admin'|'operator')를 부여한다. 읽기 라우터와 쓰기 라우터를 분리하거나, 최소한 vms/upgrade-tools·alarm-mutes·ui-settings·deep-search 등 mutation에 default-deny로 가드를 건다. authMiddleware 뒤에 라우터 단위 requireRole 미들웨어 추가가 가장 간단.

### H3. /api/remote/quick-connect·/probe 비-admin 허용 — 임의 host:port로 프록시 매핑/내부 SSH·포트 탐침(내부 피벗)
- **분류**: security · **확신도**: high
- **위치**: `server/src/routes/remote.js:39,161`
- **문제**: (SECURITY-AUDIT 기존 C5/H1, 미수정) remote.js의 대부분 라우트는 adminOnly이지만 POST /probe(39)와 POST /quick-connect(161)에는 adminOnly가 없다. quick-connect는 body.targetHost/targetPort(임의 IP:포트)로 addMapping을 만들고 provision()으로 HAProxy 설정에 배포한다(171-173) — owner가 viewer여도 가능. 이로써 포탈 프록시를 임의 내부 대상으로 향하는 오픈릴레이/피벗으로 만들 수 있다. /probe는 프록시 SSH로 ping/`</dev/tcp/${targetHost}/${targetPort}`를 실행(remote.js:50-52)해 내부 포트스캔이 가능하다(SAFE_HOST 정규식이 쉘 메타문자는 막지만 임의 host:port 자체는 허용). 발동조건: 인증 활성 + 임의 인증 계정.
- **수정안**: quick-connect·probe에 requireRole('admin')(또는 operator) 부여. targetHost를 자유 입력이 아니라 등록된 인벤토리 VM IP로만 한정(vcenterId 기준 화이트리스트). RFC1918 외부/loopback/link-local(169.254.169.254) 차단.

### H4. AD/LDAP TLS 인증서 검증 기본값 off — LDAPS MITM으로 자격증명 탈취/권한 위조
- **분류**: security · **확신도**: high
- **위치**: `server/src/auth/ad.js:30,71`
- **문제**: (SECURITY-AUDIT 기존 M3, 미수정) tlsRejectUnauthorized 기본값이 false다: ENV_DEFAULTS는 process.env.AD_TLS_REJECT_UNAUTHORIZED === 'true'(ad.js:30)이라 미설정 시 false, makeClient는 tlsOptions.rejectUnauthorized: Boolean(ad.tlsRejectUnauthorized)(71)로 그대로 전달한다. ldaps:// URL을 써도 서버 인증서를 검증하지 않으므로, AD로 향하는 경로의 중간자가 self-signed 인증서로 가로채면 사용자의 AD 비밀번호(UPN simple bind로 평문 password가 bind에 사용됨, ad.js:110)를 그대로 수집하고, memberOf 응답을 조작해 admin 역할을 부여받을 수 있다. 발동조건: AD 로그인 활성 + 검증 명시 활성(true) 미설정.
- **수정안**: 기본값을 검증 활성(true)으로 바꾸고 명시적 opt-out만 false 허용: tlsRejectUnauthorized: process.env.AD_TLS_REJECT_UNAUTHORIZED !== 'false'. saveAdConfig에서 들어오는 값도 엄격 불리언화.

### H5. 기본 관리자 비밀번호 admin123 자동 시드 — 기본 설치본 원격 즉시 탈취
- **분류**: security · **확신도**: high
- **위치**: `server/src/config.js:146, server/src/auth/auth.js:95-102`
- **문제**: (SECURITY-AUDIT 기존 C1, 부분 미수정) users.json이 없으면 username=admin, password=config.auth.defaultAdminPassword로 시드하는데 그 기본값이 'admin123'이다(config.js:146). 로그 평문 노출은 제거됐고(auth.js:97 주석/경고만), 로그인 잠금은 추가됐으나 위 X-Forwarded-For 우회 finding과 결합하면 무차별 없이도 알려진 공개 자격증명으로 즉시 로그인된다. 발동조건: DEFAULT_ADMIN_PASSWORD 미설정 + config/users.json 미생성 상태의 신규 설치.
- **수정안**: 기본 비번이 admin123이면 기동 시 강제 비밀번호 변경 플로우를 요구하거나 랜덤 비번 생성 후 최초 로그인 시 변경 강제. 최소한 admin123이 그대로면 변경 전까지 admin 외 동작을 막는 'must-change' 플래그를 users.json에 두고 모든 mutation을 거부.

### H6. HAProxy 설정 주입: addMapping targetHost 무검증 → 임의 디렉티브/백도어 frontend 삽입 (비-admin 발동 가능)
- **분류**: injection · **확신도**: high
- **위치**: `server/src/proxy/registry.js:193-215, server/src/proxy/deploy.js:14-27`
- **문제**: (SECURITY-AUDIT H6 기존, 미수정) addMapping()은 targetHost에 대해 `if(!targetHost)` 진리값 검사만 하고 문자/개행 검증이 전혀 없다. 이 값이 buildConfigBlock()에서 `server target ${m.targetHost}:${m.targetPort}` 줄로 그대로 들어가 deployToProxy가 원격 haproxy.cfg에 splice 후 `haproxy -c`로 검증·reload한다. targetHost에 개행을 넣으면(예: `1.2.3.4\n    server x 127.0.0.1:22\nlisten backdoor\n    bind *:9999\n    server a 10.0.0.1:22`) 관리되는 블록 안에 임의 listen/frontend/server 디렉티브를 주입해 백도어 릴레이/admin socket을 만들 수 있다. 발동: addMapping은 admin 라우트 POST /remote/mappings(admin)뿐 아니라 비-admin이 호출 가능한 POST /remote/quick-connect(routes/remote.js:161, adminOnly 없음)에서도 ephemeral 매핑으로 호출된다 → 임의 인증 사용자가 매핑을 생성하고, 이후 admin이 한 번 deploy하면 주입 디렉티브가 배포된다.
- **수정안**: addMapping에서 targetHost를 엄격 검증: IPv4/IPv6 또는 RFC1123 호스트명만 허용(`/^[A-Za-z0-9.:-]+$/` + 개행/공백/콤마 금지), 길이 제한. buildConfigBlock에서도 m.targetHost/targetPort를 재검증하고 개행 포함 시 거부. 추가로 quick-connect의 targetHost는 실제 인벤토리 VM IP(store.get().vms의 ipAddresses)로 제한.

### H7. 오픈릴레이/내부 피벗: quick-connect·probe가 비-admin 허용 + 임의 host:port 매핑/탐침
- **분류**: security · **확신도**: high
- **위치**: `server/src/routes/remote.js:39, server/src/routes/remote.js:161`

### H11. JSON 설정 스토어 전반의 비원자적 writeFileSync — 쓰기 중 크래시/정전 시 파일 손상 → 다음 로드에서 silent 전량 유실
